# Encryption

Aegis uses AES-256-GCM field-level encryption to protect all sensitive data stored in PostgreSQL. This document covers what is encrypted, how encryption works, key management, and key rotation.

## What is Encrypted

All sensitive data stored in PostgreSQL is encrypted with AES-256-GCM before being written to the database:

- **Credentials** -- API tokens, OAuth refresh tokens, passwords for third-party services (Plaid, Canvas, Garmin, etc.)
- **Financial account identifiers** -- Plaid access tokens, account IDs
- **Health data** -- Raw health metrics from Garmin and Apple Health

Each encrypted field uses:
- A unique random nonce (96 bits) per encryption operation
- Authenticated Additional Data (AAD) that includes the user ID and field context, binding the ciphertext to its intended row
- The `ENCRYPTION_MASTER_KEY` from `.env` (256-bit key, hex-encoded)

## What is NOT Encrypted

The following are stored in plaintext because they are not sensitive or because encryption would prevent useful queries:

- Transaction amounts and dates (needed for SQL aggregation)
- Transaction categories and merchant names
- Assignment titles and due dates
- Audit log entries (integrity protected by hash chain, not encryption)
- LLM usage records (token counts, cost calculations)
- Content draft text (not considered PII)

## How It Works

The encryption implementation lives in `data-api/app/security/encryption.py`.

### Encryption Flow

1. A plaintext value arrives (e.g., a Canvas API token)
2. A random 96-bit nonce is generated
3. AAD (Authenticated Additional Data) is constructed from the user ID and resource context
4. AES-256-GCM encrypts the plaintext using the master key, nonce, and AAD
5. The nonce + ciphertext + GCM tag are stored together in the database column

### Decryption Flow

1. The stored blob is split into nonce, ciphertext, and tag
2. AAD is reconstructed from the user ID and resource context
3. AES-256-GCM decrypts and verifies the tag
4. If the AAD does not match (e.g., someone copied the ciphertext to a different row), decryption fails

### Why AAD Matters

AAD (Authenticated Additional Data) prevents a class of attacks where an attacker with database access could copy encrypted values between rows. Because the AAD includes the user ID and field context, the ciphertext is bound to its specific row. Moving it to another context causes decryption to fail.

## Key Management

### The Master Key

- Stored in `.env` as `ENCRYPTION_MASTER_KEY`
- 256-bit key encoded as a 64-character hex string
- Generated by `bootstrap.sh` using `openssl rand -hex 32`
- Never committed to version control
- Injected into the data-api container via Docker Compose environment variables

### Generating a New Key

```bash
openssl rand -hex 32
```

This produces a 64-character hex string suitable for `ENCRYPTION_MASTER_KEY`.

## Key Rotation

The encryption key can be rotated using `infrastructure/scripts/rotate-secrets.sh`:

```bash
ROTATE_ENCRYPTION_KEY=true ./infrastructure/scripts/rotate-secrets.sh
```

This process:

1. Generates a new encryption key
2. Re-encrypts all stored credentials with the new key
3. Updates `.env` with the new key
4. Restarts all services

**Warning:** Rotating the encryption key without re-encrypting credentials will make them permanently unreadable. Always use the rotation script rather than manually changing the key.

## SOPS + age for Secret Files

For version-controlled secret files (not database fields), Aegis uses SOPS + age:

- Encrypted files are stored in `secrets/*.enc.yaml`
- Decrypted at deploy time by `deploy.sh`
- Cleaned up on exit (decrypted files are temporary)
- `.gitignore` excludes all unencrypted `.yaml` files in `secrets/`

This is separate from the AES-256-GCM field encryption -- SOPS handles files at rest in the repository, while AES-256-GCM handles fields at rest in the database.

## Security Properties

| Property | Status |
|----------|--------|
| Encryption algorithm | AES-256-GCM (authenticated encryption) |
| Nonce reuse prevention | Unique random 96-bit nonce per operation |
| Ciphertext binding | AAD includes user ID + field context |
| Key strength | 256-bit (64 hex characters) |
| Key storage | Environment variable, never in version control |
| Tag verification | Automatic with GCM mode (detects tampering) |
| Forward secrecy | Not applicable (symmetric encryption) |
